"Add a fully featured Procurement Module to the Zentryx app. This is a brand new module — do not modify any existing modules, components, routes, or UI patterns. Match the existing design system exactly: same Tailwind classes, same card styles, same table layouts, same light/dark mode handling, same sidebar navigation pattern, same Kanban, same export logic, same notification system.
Context about the existing app: The app is a React + Vite + TailwindCSS + Zustand frontend with an Express 5 backend, PostgreSQL database via Drizzle ORM, Zod v4 validation, JWT auth, and an OpenAPI spec with Orval-generated React Query hooks. The monorepo structure is:
* Frontend: artifacts/rd-intelligence/
* Backend: artifacts/api-server/
* DB schema: lib/db/
* OpenAPI spec: lib/api-spec/
* Generated hooks: lib/api-client-react/
Study the Sales Force module (/sales-force) as the reference pattern for structure, layout, tabs, and style — the Procurement module should feel like a sibling of Sales Force, not a foreign addition. Also study the existing notification system, activity log, and how the Forecast tab sends notifications to staff, as Procurement will integrate with all of these.

STEP 1 — DATABASE SCHEMA
In lib/db/schema.ts, add the following new tables using Drizzle ORM. Follow the exact same column conventions (uuid primary keys, timestamps, snake_case) used in the existing schema:
1. vendors — vendor registry
    * id (uuid, pk), name, category (Ingredients / Packaging / Equipment / Services / Logistics / Other), contactName, contactEmail, contactPhone, country, address, paymentTerms (Net15 / Net30 / Net60 / COD), currency (NGN / USD / EUR / GBP), rating (integer 1–5), status (Active / Inactive / Blacklisted), notes, createdAt, updatedAt
2. purchase_requests — internal purchase requests raised by any staff member
    * id, title, description, requestedBy (FK → users), department (FK → departments), vendorId (FK → vendors, nullable), category (same as vendor category), priority (Low / Medium / High / Critical), status (Draft / Pending Approval / Approved / Rejected / Cancelled / Converted to PO), estimatedAmount, currency, requiredByDate, justification, attachmentUrl, createdAt, updatedAt
3. purchase_request_approvals — multi-level approval chain per request
    * id, purchaseRequestId (FK), approverId (FK → users), level (integer: 1, 2, 3), status (Pending / Approved / Rejected), comment, decidedAt, createdAt
4. purchase_orders — generated from approved purchase requests
    * id, poNumber (auto-generated: PO-YYYY-XXXX), purchaseRequestId (FK, nullable), vendorId (FK → vendors), raisedBy (FK → users), status (Draft / Sent to Vendor / Acknowledged / In Transit / Partially Received / Received / Closed / Cancelled), totalAmount, currency, paymentStatus (Unpaid / Partially Paid / Paid), paymentDue, deliveryAddress, deliveryDue, notes, createdAt, updatedAt
5. purchase_order_items — line items per PO
    * id, purchaseOrderId (FK), description, quantity, unit (kg / litres / units / cartons / bags / packs), unitPrice, totalPrice, productType (match existing Zentryx product types), notes
6. purchase_order_receipts — goods received notes (GRN)
    * id, purchaseOrderId (FK), receivedBy (FK → users), receivedAt, quantityReceived, condition (Good / Damaged / Partial), notes
7. vendor_performance — auto-calculated or manually rated per PO completion
    * id, vendorId (FK), purchaseOrderId (FK), deliveryScore (1–5), qualityScore (1–5), communicationScore (1–5), overallScore (computed average), reviewedBy (FK → users), notes, createdAt
After adding schema, run pnpm --filter @workspace/db run push to apply.

STEP 2 — BACKEND API ROUTES
In artifacts/api-server/src/routes/, create procurement.ts. Register it in the main router as /api/procurement. Build the following endpoints, using the same middleware pattern (JWT auth, Zod validation) as existing routes:
Vendors:
* GET /procurement/vendors — list all, with filters: category, status, country, rating
* POST /procurement/vendors — create vendor
* GET /procurement/vendors/:id — get vendor detail with their PO history and performance scores
* PUT /procurement/vendors/:id — update vendor
* DELETE /procurement/vendors/:id — soft delete (set status to Inactive)
Purchase Requests:
* GET /procurement/requests — list all, filters: status, priority, department, requestedBy, dateRange
* POST /procurement/requests — create new request (status: Draft)
* GET /procurement/requests/:id — get with approval chain and comments
* PUT /procurement/requests/:id — update (only if Draft or Rejected)
* POST /procurement/requests/:id/submit — change status to Pending Approval, trigger notification to approvers
* POST /procurement/requests/:id/approve — approver approves their level; if final level approved, status → Approved; trigger notification to requester
* POST /procurement/requests/:id/reject — approver rejects with required comment; status → Rejected; notify requester
* POST /procurement/requests/:id/convert-to-po — creates a Purchase Order from an Approved request
Purchase Orders:
* GET /procurement/orders — list all POs, filters: status, vendorId, paymentStatus, dateRange
* POST /procurement/orders — create PO directly (without PR)
* GET /procurement/orders/:id — get PO with line items, receipts, and vendor info
* PUT /procurement/orders/:id — update PO (only if Draft)
* POST /procurement/orders/:id/send — mark as Sent to Vendor, notify vendor contact via the existing notification + mailto system
* POST /procurement/orders/:id/receive — log goods received (creates purchase_order_receipts entry), updates PO status
* GET /procurement/orders/export — export filtered POs to CSV and XLSX using exceljs (same pattern as Sales Force export)
Analytics:
* GET /procurement/analytics — return: total spend by category (current month, YTD), PO status distribution, top 5 vendors by spend, average PO cycle time (created → received), pending approval count, overdue deliveries count, monthly spend trend (last 12 months)
Notifications integration:
* When a PR is submitted for approval: notify all users with Manager or Admin role in the relevant department
* When a PR is approved or rejected: notify the requester
* When a PO is marked Received: notify the original requester and the Finance department if it exists
* When a delivery is overdue (deliveryDue has passed, status not Received): flag in analytics — do not auto-send notifications, but surface in the dashboard
Add all new endpoints to lib/api-spec/openapi.yaml following the existing spec format, then run pnpm --filter @workspace/api-spec run codegen to regenerate the React Query hooks in lib/api-client-react/.

STEP 4 — SYNC WITH EXISTING MODULES
These integrations must work without modifying the existing modules' own code — add to the Procurement module only:
1. Sales Force → Procurement sync: The existing "Notify Procurement" button in the Sales Forecast tab already sends a notification. In the Procurement Requests tab, add a filter "Source: From Sales Forecast" so procurement staff can see requests that were triggered from the Sales team. When a Sales forecast notification arrives (type: System, containing forecast data), surface it as a draft Purchase Request pre-filled with the product and volume data from the notification payload.
2. Notification system: Use the existing POST /api/notifications endpoint and the existing notification bell UI — do not build a new one. All procurement events (PR submitted, approved, rejected, PO received, overdue) go through the same notifications table and display in the existing notification dropdown.
3. Activity log: Write all procurement actions (vendor created, PR submitted, PO raised, goods received) to the existing activity_logs table using the same format as other modules. These will then appear in the existing Activity feed automatically.
4. Team Directory: When selecting approvers for a purchase request approval chain, use the existing GET /api/users endpoint to fetch team members. Display them with the same avatar initials + name + role format used in Team Directory.
5. Weekly Activities: Add "Procurement" as a valid activity type in the Weekly Activities tracker's Product Type or Status dropdowns — this allows procurement staff to log their weekly procurement activities in the same tracker other teams use.
